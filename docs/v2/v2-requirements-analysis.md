# Beehive Blog v2 需求分析

## 1. 产品定义

Beehive Blog v2 的定位是：

**个人博客 + AI 协作创作平台 + 面向智能体的个人知识中台**

它不是单纯的博客系统，也不是单纯的知识库，而是一个同时服务以下三类对象的平台：

- 你自己
- 普通访客与读者
- 外部智能体应用

## 2. v2 要解决的核心问题

当前 v2 需要优先解决的问题有三类：

### 2.1 工程问题

- v1 代码结构混乱
- 功能边界不清晰
- 内容、展示、同步、后台逻辑耦合过深

### 2.2 内容问题

- 你的知识沉淀分散
- 博客、项目、经历、笔记、反思之间缺乏统一结构
- 内容难以复用、检索和总结

### 2.3 表达问题

- 个人经历难以结构化表达
- 面向读者的文章和面向自己的原始笔记没有被正确区分
- 智能体无法安全、稳定地读取与协作

## 3. 目标用户

v2 面向三类核心用户。

### 3.1 你自己

作为知识生产者、整理者、审阅者和最终管理员。

### 3.2 普通访客与读者

他们主要消费公开内容，包括：

- 博客文章
- 个人简介
- 项目经历
- 技术专题
- 知识地图
- 作品集

### 3.3 外部智能体应用

例如：

- Hermes
- OpenClaw
- 其他基于 MCP、Skill、Agent API 的客户端

它们主要消费结构化知识，并辅助生成草稿、摘要、周报和关联关系。

## 4. 核心内容类型

第一阶段明确支持以下内容类型：

- 文章
- 笔记
- 项目
- 个人经历
- 反思 / 洞察
- 作品集
- 附件资料
- 时间线事件

## 5. 内容模型的基本理解

### 5.1 文章

面向读者发布的正式内容。

特点：

- 结构更完整
- 适合公开展示
- 更偏博客表达

### 5.2 笔记

偏原始、碎片、私密或半私密。

特点：

- 更偏向给自己和 AI 使用
- 可以部分开放给读者，但默认不等同于正式文章
- 适合作为后续文章、总结和知识提炼的原材料

### 5.3 项目

用于承载项目背景、目标、方案、技术、结果、作品和关联文章。

### 5.4 个人经历

以时间线为主轴组织。

第一阶段要求：

- 个人经历按时间线组织
- 可以和项目、文章、反思、作品集进行关系绑定

### 5.5 时间线事件

作为经历的最细粒度节点，用于记录：

- 某段时间做了什么
- 发生了什么变化
- 有哪些对应输出和反思

## 6. 关键关系设计

### 6.1 项目与文章

项目和文章不强制父子结构，采用关系绑定。

这意味着：

- 一个项目可以关联多篇文章
- 一篇文章也可以关联多个项目
- 内容之间通过关系表组织，而不是目录层级写死

### 6.2 经历与其他内容

经历作为时间主轴，可以关联：

- 项目
- 文章
- 笔记
- 反思
- 作品
- 附件

### 6.3 笔记到文章的演进

v2 应支持以下演化路径：

- 笔记 -> 摘要
- 笔记 -> 草稿
- 笔记 -> 文章
- 多篇笔记 -> 周报 / 阶段总结 / 技术专题

## 7. AI 在系统中的角色

第一阶段，AI 的角色是“辅助创作与整理”，不是“自动发布者”。

AI 需要支持以下能力：

- 搜索知识库
- 阅读指定内容
- 生成草稿
- 生成摘要
- 输出周报 / 阶段总结
- 建立内容关联
- 整理标签、摘要和关联关系

AI 不应具备的能力：

- 直接发布正式内容
- 越过人工审阅
- 读取未授权的私密内容

## 8. 公开站目标

公开站第一阶段重点展示：

- 博客文章
- 个人简介
- 项目经历
- 技术专题
- 知识地图
- 搜索
- 作品集
- 联系方式

说明：

- 知识地图可以先保留信息架构接口，功能上后期增强
- 后台优先于公开站，但公开站仍然是第一阶段必须覆盖的场景

## 9. Studio / 后台目标

Studio 第一阶段重点解决：

- 内容编辑
- 版本管理
- AI 草稿审阅
- 标签和关系维护
- 发布管理
- 个人经历整理
- 附件管理

这意味着 v2 的后台不是传统 CMS，而是：

**内容工作台 + 知识整理台 + AI 协作审阅台**

## 10. 搜索与智能体接入目标

第一阶段与 AI 相关的目标是：

- 先做好站内搜索
- 为 Hermes / OpenClaw 等外部智能体提供可控的知识读取能力
- 私密内容和公开内容必须严格隔离

这说明 v2 第一阶段应优先建设：

- 内容标准化
- 搜索与检索
- 权限隔离

RAG、问知识库和复杂 AI 协作可以建立在这些能力之上逐步增强。

## 11. 第一阶段最低可交付目标

v2 第一阶段的最低目标已经比较明确：

- 能发文章
- 能搜索
- 能管理经历
- 能管理项目
- 能管理文章

在此基础上，建议补上：

- 基础笔记管理
- 版本管理
- 附件管理
- 内容关系管理

## 12. 简化权限模型建议

你提到权限系统需要再简化，我建议 v2 第一阶段不要把权限设计成很多零散开关，而是拆成 3 个维度。

### 12.1 用户角色

第一阶段建议只保留 4 类主体：

- `guest`
- `member`
- `owner`
- `agent`

解释：

- `guest`：未登录访客
- `member`：普通登录用户
- `owner`：你自己，拥有全部后台与审阅权限
- `agent`：通过系统授权接入的智能体主体

### 12.2 内容状态

第一阶段建议统一为：

- `draft`
- `review`
- `published`
- `archived`

解释：

- `draft`：编辑中，不对外发布
- `review`：待你审核
- `published`：正式发布
- `archived`：归档，不再作为当前内容展示重点

### 12.3 可见范围

第一阶段建议统一为：

- `public`
- `member`
- `private`

解释：

- `public`：访客可见，可进入公开搜索
- `member`：登录用户可见，不进入公开搜索
- `private`：仅你自己可见，不进入公开搜索

### 12.4 AI 读取策略

不要把 AI 权限混在可见范围里，单独保留一个简单字段即可：

- `ai_access = denied`
- `ai_access = allowed`

这样好处很明显：

- 模型清晰
- 权限判断简单
- 避免出现很多难维护的组合状态

### 12.5 第一阶段推荐权限规则

推荐默认规则如下：

- `draft` 默认 `private + ai_access=denied`
- `review` 默认 `private + ai_access=denied`
- `published + public` 可进入公开站和公开搜索
- `published + member` 仅登录用户可见，不进入公开搜索
- `published + private` 仅你自己可见
- 任意 `private` 内容不得进入公开搜索
- 任意 `ai_access=denied` 内容不得被智能体读取
- 私密经历默认 `private + ai_access=denied`

### 12.6 为什么这样设计

这套模型比“仅自己可见、仅 AI 可读、登录可见、草稿不可 AI 读取、私密经历不能公开搜索”这些规则更容易落地。

因为它把问题拆开了：

- 谁能看：角色 + 可见范围
- 内容处于什么阶段：状态
- AI 能不能读：单独控制

这比把所有条件混在一个权限字段里更稳定。

## 13. 第一阶段建议暂不做复杂权限

为了控制复杂度，第一阶段不建议立即支持：

- 多人协同编辑权限
- 细粒度字段级权限
- 多团队、多空间权限
- 多审核人流转

原因很简单：

- 你当前审核人只有自己
- 产品重点在内容和知识沉淀，不在复杂组织协作

## 14. 评论、注册与 Agent 边界补充

### 14.1 评论策略

第一阶段评论默认不做人工审核拦截。

建议方式：

- 评论提交后可直接展示
- 同时接入基础审核接口或内容检查模块
- 审核模块主要承担风险检查、垃圾内容识别和后续治理能力

这意味着第一阶段更偏向：

- 简化互动流程
- 降低使用门槛
- 预留后续审核能力，而不是一开始把评论流转做重

### 14.2 `member` 内容的 AI 读取

第一阶段允许：

- `visibility=member` 的内容被 AI 读取

前提是：

- `ai_access=allowed`

因此最终判定逻辑应是：

- 先看内容可见范围
- 再看 `ai_access`

### 14.3 `agent` 身份边界

第一阶段将 `agent` 视为统一的系统授权主体。

这意味着：

- 不区分 Hermes / OpenClaw / 其他客户端的独立角色模型
- 先由系统统一发放和管理 Agent 访问能力
- 后续如有需要，再细分 Agent Provider 或 Agent Client 身份

### 14.4 注册策略

第一阶段开放普通用户注册。

这意味着 v2 第一阶段除了 owner 和 agent 外，还需要完整考虑：

- `guest -> member` 转化路径
- 登录用户评论
- member 可见内容
- 公开内容与登录后内容的边界

## 15. 第一阶段功能范围建议

### 必做

- 用户登录
- 用户注册
- owner 后台
- 内容创建与编辑
- 文章、笔记、项目、经历、时间线事件管理
- 标签和关系管理
- 内容版本管理
- 附件管理
- 发布管理
- 搜索

### 建议做

- AI 草稿生成
- AI 摘要生成
- 周报 / 阶段总结生成
- MCP 基础读取接口

### 后期做

- 知识地图可视化
- 更复杂的 RAG 问答
- 多人协作与高级权限
- 更复杂的推荐与图谱分析

## 16. 当前阶段仍待确认的问题

虽然整体方向已经比较清楚，但后续进入产品设计前，还需要继续确认这些问题：

1. 公开站第一阶段是否必须包含知识地图页面，还是只预留接口
2. 是否要在第一阶段开放用户个人主页
3. 评论体系是否需要支持楼中楼与通知联动
4. AI 草稿是否允许直接引用 `member` 内容作为输出来源
5. Agent 调用是否需要按 API key / app / workspace 做独立审计
6. 是否需要“内容被 AI 使用记录”可视化页面

## 17. 当前结论

到目前为止，v2 第一阶段已经可以清晰定义为：

**一个后台优先、公开站同步推进的个人知识平台，支持文章、笔记、项目、经历与时间线内容的统一管理，并通过简化权限模型和受控 AI 接入能力，为后续搜索、智能体学习和协作创作打基础。**
