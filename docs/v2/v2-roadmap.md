# Beehive Blog v2 开发路线规划

## 1. 总体策略

v2 采用“先文档、后骨架、再最小可用、最后增强”的路线。

目标不是一次性做完所有能力，而是按依赖顺序稳步推进。

## 2. Phase 0：设计冻结

目标：

- 固化需求分析
- 固化产品设计
- 固化服务边界

交付物：

- `v2-requirements-analysis.md`
- `v2-product-design.md`
- `v2-roadmap.md`
- `v2-domain-model.md`
- `v2-api-design.md`

## 3. Phase 1：基础骨架

目标：

- 建立微服务仓库结构
- 初始化 go-zero 工程骨架
- 建立基础部署与配置体系

建议交付：

- `gateway`
- `identity-service`
- `content-service`
- `search-service`
- `indexer-worker`
- 基础 docker 编排

完成标准：

- 各服务可以本地启动
- 网关可转发
- 基础配置、日志、鉴权骨架可用

## 4. Phase 2：内容平台最小闭环

目标：

- 先让系统能真正管理核心内容

范围：

- 注册 / 登录
- owner 管理后台
- 文章管理
- 笔记管理
- 项目管理
- 经历管理
- 时间线事件管理
- 标签与关系管理
- 附件管理
- 状态流转
- 版本管理

完成标准：

- 你能在 Studio 里完整管理核心内容

## 5. Phase 3：公开站最小闭环

目标：

- 让内容可以被读者消费

范围：

- 首页
- 文章列表 / 详情
- 项目列表 / 详情
- 经历 / 时间线页
- 作品集页
- 关于页
- 登录 / 注册
- 评论

完成标准：

- 可以替代现有博客的基础展示职责

## 6. Phase 4：搜索能力上线

目标：

- 打通站内搜索与内容索引

范围：

- 基础索引同步
- PostgreSQL FTS 或 Meilisearch
- 搜索 API
- 搜索页
- 高亮摘要
- 类型 / 标签 / 时间过滤

完成标准：

- 你和访客都能依赖搜索找到内容

## 7. Phase 5：AI 协作第一阶段

目标：

- 让 AI 成为受控助手

范围：

- AI 摘要
- AI 草稿生成
- 周报 / 阶段总结生成
- AI 草稿中心
- 审阅后发布

完成标准：

- AI 输出必须带来源
- AI 输出必须先经过你审核

## 8. Phase 6：智能体接入第一阶段

目标：

- 让外部智能体读取知识库

范围：

- MCP 基础 server
- `search_content`
- `read_content`
- `summarize_content`
- 受控 `write_draft`

完成标准：

- Hermes / OpenClaw 类客户端可以安全读取公开或授权内容

## 9. Phase 7：增强阶段

范围：

- 知识地图可视化
- 混合检索与向量搜索
- 更复杂的 RAG
- 更强的引用和审计
- 高级内容推荐
- Agent 使用记录与审计后台

## 10. 当前优先级建议

如果按实际落地价值排序，我建议：

1. 文档定稿
2. 内容模型
3. 服务骨架
4. Studio 最小闭环
5. 公开站最小闭环
6. 搜索
7. AI 草稿与摘要
8. MCP / Agent 接入

## 11. 第一阶段里程碑定义

可以把 v2 第一阶段定义为：

**完成一个可用的个人知识平台基础版，支持文章、笔记、项目、经历、时间线、搜索和基础公开展示。**

达到这个里程碑后，再进入更强的 AI 协作阶段。
