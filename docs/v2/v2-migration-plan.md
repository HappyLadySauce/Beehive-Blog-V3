# Beehive Blog v2 迁移计划

## 1. 目标

本文件定义从当前已归档状态迁移到 v2 的策略。

目标不是兼容运行 v1，而是：

- 保留可复用知识
- 保留可迁移数据
- 清晰进入 v2 的新仓库和新服务结构

## 2. 迁移原则

### 2.1 迁移内容，不迁移旧架构

v1 的价值在于：

- 需求沉淀
- 文档沉淀
- 历史内容
- 数据经验

不在于：

- 继续沿用旧目录结构
- 继续沿用旧服务边界
- 继续沿用 Hexo 中心的内容组织方式

### 2.2 迁移分三层

建议把迁移拆成：

- 文档迁移
- 数据迁移
- 功能迁移

### 2.3 先迁文档和数据，再迁功能

顺序必须明确：

1. 固化 v2 文档
2. 设计新模型
3. 迁移内容数据
4. 重建服务

不要先重写代码，再想着如何搬历史内容。

## 3. 当前资产分类

## 3.1 已保留资产

当前应继续保留：

- `docs/v2/`
- 根目录的 v2 架构和搜索设计文档
- `docs/archive-v1/` 或等价归档资料
- `docker/`

## 3.2 可迁移资产

建议识别为“未来可导入”的资产：

- 历史文章内容
- 历史笔记或 Markdown
- 标签与分类
- 附件
- 时间相关资料
- 项目资料
- 个人经历素材

## 3.3 不迁移资产

建议不迁移：

- v1 的路由结构
- v1 的 Hexo 渲染耦合逻辑
- v1 的页面组织方式
- v1 的 Swagger 生成物
- v1 的前后端项目结构

## 4. 文档迁移策略

第一阶段文档已经在推进中。

建议最终形成两类文档：

### 4.1 v2 正式文档

保留在：

- `docs/v2/`

### 4.2 v1 存档文档

保留在：

- `docs/archive-v1/`

用途：

- 追溯历史需求
- 参考旧实现经验
- 辅助人工迁移数据

## 5. 数据迁移策略

## 5.1 迁移目标对象

第一阶段建议优先迁移：

- 文章
- 笔记素材
- 项目资料
- 个人经历资料
- 标签
- 附件

## 5.2 迁移目标模型

统一迁移进入：

- `content_items`
- `content_revisions`
- `content_tags`
- `content_relations`
- `attachments`
- `content_attachments`

## 5.3 历史博客文章迁移

建议映射：

- v1 博客文章 -> `content_items(type=article)`
- 正文 -> `content_revisions.body_markdown`
- 标签 -> `tags + content_tags`
- 封面 / 图片 / 文件 -> `attachments + content_attachments`

## 5.4 原始素材与零散记录迁移

建议映射：

- 零散 Markdown
- 草稿
- 个人记录

迁移为：

- `content_items(type=note)`

不要强行在迁移阶段把所有零散内容直接整理成 article。

## 5.5 项目与经历迁移

如果历史数据中已经存在明显的项目或阶段信息：

- 项目 -> `type=project`
- 经历阶段 -> `type=experience`
- 关键节点 -> `type=timeline_event`

如果历史资料还不够结构化：

- 先导入为 note
- 后续再在 Studio 中人工整理绑定

## 5.6 评论迁移

第一阶段可选。

建议：

- 若历史评论价值不高，可不迁移
- 若评论承载有效讨论，再设计迁移脚本

## 6. 附件迁移策略

附件迁移必须注意两件事：

- 文件本体迁移
- 内容引用关系迁移

建议流程：

1. 统一整理旧附件目录
2. 为每个附件生成 checksum
3. 导入 `attachments`
4. 建立 `content_attachments`

## 7. slug 迁移策略

v2 建议全局唯一 `slug`。

迁移时要考虑：

- 原有 slug 是否冲突
- 原有路径是否规范

建议：

- 尽可能保留历史公开文章 slug
- 对冲突内容生成新 slug，并记录映射表

可选增加：

- `legacy_slug_redirects`

用于后续做重定向兼容。

## 8. 迁移阶段划分

## Phase A：静态资产盘点

目标：

- 列出所有历史内容来源
- 明确哪些内容迁移、哪些仅归档

产出：

- 内容清单
- 附件清单
- 旧 slug 清单

## Phase B：结构映射确认

目标：

- 确认 v1 数据如何映射到 v2 表结构

产出：

- 字段映射表
- 内容类型映射表

## Phase C：导入脚本

目标：

- 编写一次性导入脚本

建议输出：

- `scripts/migrate/import_content.go` 或等价工具

## Phase D：人工整理

目标：

- 在 Studio 中补齐项目、经历、关系

说明：

很多高价值知识结构无法完全自动迁移，必须接受人工整理阶段的存在。

## 9. 功能迁移策略

功能迁移不应按“模块名”做，而应按“业务能力”做。

### 第一优先级

- 内容创建与编辑
- 发布
- 搜索
- 项目和经历管理

### 第二优先级

- 评论
- AI 摘要
- AI 草稿

### 第三优先级

- MCP
- 复杂检索
- 知识地图

## 10. 风险点

## 10.1 风险：历史内容结构不一致

处理建议：

- 不强求全自动
- 先导入为 note，再人工升级

## 10.2 风险：slug 冲突

处理建议：

- 提前生成映射表
- 保留重定向能力

## 10.3 风险：附件丢失或引用断裂

处理建议：

- 先做附件清单和 checksum
- 迁移脚本必须记录失败项

## 10.4 风险：搜索与 AI 读取到不该读的历史内容

处理建议：

- 所有迁移内容默认 `private + ai_access=denied`
- 迁移后由 owner 分批审核和发布

## 11. 当前建议

迁移执行顺序建议固定为：

1. 完成 v2 文档
2. 搭 v2 服务骨架
3. 建 v2 数据库
4. 写迁移脚本
5. 导入历史内容为 draft/private
6. 人工整理并逐步发布

## 12. 当前结论

v2 的迁移策略应当是：

**迁移历史内容和知识资产，不迁移旧架构；所有历史数据先安全落库，再逐步整理为 v2 的结构化知识。**
